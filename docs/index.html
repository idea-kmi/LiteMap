Intro

This is a really quick overview of the site structure with some instructions further down on setting up the database and config file.

NOTE: The LiteMap code base is released under the LGPL license: http://www.fsf.org/licensing/licenses/lgpl.html.
It includes third party code which should all have licenses which are compatible with LGPL (see /core/lib and /ui/lib folders to view third party libraries used)

System Requirements

The LiteMap code has been developed and tested on a Red Hat Enterprise Linux server using:

Apache 2.4.6.

Please add the following in your Apache config:

Options All -Indexes -MultiViews
RewriteEngine on
AllowOverride FileInfo AuthConfig
LiteMap has embeddable iframes as part of its infastructure. Therefore LiteMap needs to have permissions set so the browsers allow LiteMap pages to be embedded in thrid party websites.

If you have .htaccess files disabled (AllowOverride none), it will not affect the running of the site.
Our .htaccess file has the following: 

AddOutputFilterByType DEFLATE text/plain
AddOutputFilterByType DEFLATE text/html
AddOutputFilterByType DEFLATE text/xml
AddOutputFilterByType DEFLATE text/css
AddOutputFilterByType DEFLATE text/javascript
AddOutputFilterByType DEFLATE text/x-js
AddOutputFilterByType DEFLATE application/xml
AddOutputFilterByType DEFLATE application/xhtml+xml
AddOutputFilterByType DEFLATE application/rss+xml
AddOutputFilterByType DEFLATE application/javascript
AddOutputFilterByType DEFLATE application/x-javascript
AddOutputFilterByType DEFLATE application/json
AddOutputFilterByType DEFLATE application/ld+json
AddOutputFilterByType DEFLATE image/svg+xml
AddOutputFilterByType DEFLATE font/truetype
AddOutputFilterByType DEFLATE font/opentype
AddOutputFilterByType DEFLATE application/vnd.ms-fontobject


#RewriteEngine on
#RewriteCond %{REQUEST_URI} !/index-maintenance.php$
#RewriteRule $ /index-maintenance.php [R=302,L]

ErrorDocument 404 /404-error-page.php
ErrorDocument 403 /403-error-page.php

So we use it to compress files to speed up the website and for having site specific 404 and 403 pages.
We also uncomment the 'Rewrite' lines if we want to put the site into maintenace mode and display a holding page to the user (index-maintenace.php)

MySQL

LiteMap uses MySQL as the database at present. So you will need to install MySQL on your server. We have developed and tested on MySQL (MariaDB): 5.5. The database layer of the code has been abstracted out somewhat to make it eaiser for a developer to extend the LiteMap code base to use a different database.

Email

LiteMap uses emails to validate account setups, tell users their registration is successful, send out email digests etc. So your server should be able to do emailing if you want these to be sent. You can switch off emailing with a setting in the config file if required (see Config Settings, below), but then all accounts would need to be setup and validated by hand.

Crons

To send out the digest emails you will need to setup some cron jobs. An example is shown below. To use the lines below you need to add the file system path to your LiteMap code where indicated and the domain name to run then against (as LiteMap can be multi domain):

0 1 * * 4 php path to code/core/cron/cron-follow-weekly.php domain name

0 1 25 * * php path to code/core/cron/cron-follow-monthly.php domain name

0 2 * * * php path to code/core/cron/cron-follow-daily.php domain name

0 * * * * php path to code/core/cron/cron-follow-hourly.php domain name

0 0 15 * * php path to code/core/cron/cron-recent-activity.php domain name


A Quick What's Where in the 'src' folder

The top level of the file structure has the main base config.php file, (see below for more details on the domain level config file Config Settings) and the setup.php file.

This level also has the files for the main pages of the site. These are:

builder.php - this is the web bookmarklet landing page,
chats.php - the viewing chat pages,
explore.php - for viewing a given entry,
explore-slim.php - for viewing a given entry when on the map page,
group.php - for viewing a group page,
index.php - for viewing the main site page/starting point,
map.php - for a viewing given map,
knowledgetrees.php - for viewing a knowledge tree on the explore page,
networkgraph.php - for viewing a network graph on the explore page,
search.php - for viewing searhes,
user.php - for viewing a given user

The majority of these top-level page should probably move to ui/pages in the future.

The Folders in 'src':

  • api - This folder holds the LiteMap services file that processes incoming service request, (see core/apilib.php for the actual guts).
  • core - This is where the server side code resides.
    • cron - This folder has various files that run the cron jobs.
    • database - This holds the database manager and the sql statements. There is a README.txt file in there with more details.
    • datamodel - This holds that classes that hold that database record information when it is loaded. There are classes for all the mail items types, like user, node, linktype etc...
    • formats - This holds the files that process the api return data into the correct format, e.g. json, xml etc..
    • io - This folder holds the code to handle the import options in LiteMap: Currently just from Catalyst and its Catalyst Interchange Format.
    • lib - This holds all the external libraries that LiteMap is using in the server side code.
  • images - This is where most of the graphics for the site are. Some are also in the themes (see below)
  • install - This folder has the sql file needed to create a LiteMap database and the script to install the default data. See Setting up an LiteMap below.
    Once you have finished the install process remove this folder from your website.
  • language - This folder holds the language files for all the interface text. By changing the $CFG->language property (see below) you can override the default language of 'en' and add a different folder and then convert all the language files to another language if you want to. For more details on how this works see Language below.
    • en - This holds the default English language version of all interface text shown to the user. Please never remove this folder.
  • sites - There is a README.txt file with more details
    • default - This is where you setup a sinlge instance or the base instance of the site. There is a README.txt file there with more details.
    • multi - This is where you setup multiple instance of the site for different domains, but sharing the codebase. There is a README.txt file there with more details.
  • themes - This is where you can setup alternative themes. We supply the Default theme. It has a sub-folder for images and one for the stylesheets. Please see the README.txt file that is in the themes folder for more details about setitng up themes.
  • ui - This folder has most of the client side code e.g. header, footer, homepage etc... etc.
    • admin - This folder has various files for the Administrator option pages. If you are a user who has the 'IsAdministrator' field in the Users table set to 'Y', you will see an extra menu item when logged in at the top of the screen called 'admin' which will take you to the Administrator options.
    • embed - This folder has the embed file for embedding readonly and editable maps in third party sites.
    • explore - This holds that classes that create the item explore pages, like the slim version on maps and the full version on the explore.php page. It also has the chat pages code.
      • chat - The code for the chats pages.
      • slim - The code for the details divs for nodes in a map.
      • widget - The code for the explore.php pages for items.
    • lib - This holds all the external libraries that the LiteMap is using in the client side code.
    • movies - This holds all the help movies.
    • networkmaps - This holds the files that create all the visualisations used on the Dashboards and also the explore page network pages. It also holds the files that call CIDashboard for the map level visualisation pages.
    • pages - This holds the secondary page files, like login and logout, register, conditionsofuse etc..
    • popups - This holds the files that are shown in popup windows pages, like the add and edit forms, activity viewer pages etc..
    • stats - This holds internally created dashboard code for system, group and map level dashboards.

Setting up a LiteMap website

  1. Creating the Database:
    1. Create an empty database for the LiteMap to use in your MySQL installation, making sure that the database character set and collation are UTF-8, e.g.
      CREATE DATABASE dbname CHARACTER SET utf8 COLLATE utf8_general_ci;

      If you do not want to use utf8, you will need to edit the mysql.sql file before you run it in c. below, as the table creation statements are using utf8.
    2. Create a MySQL user for LiteMap to use.
    3. Load the database schema from /install/mysql.sql into your new LiteMap Database to create all the required tables.
  2. Adding the Default Data:
    1. Edit /install/adddefaultdata.php. You must fill in the following data in the file:
      • $dirAddress = the file path to where your website code resides on the server (ending in '/')
      • $databaseaddress = the address of your mysql database - usually 'localhost';
      • $databaseuser = the username of your mysql user for LiteMap, as created in step 1b above;
      • $databasepass = the password of your mysql user for LiteMap, as created in step 1b above;
      • $databasename = the name of your mysql database for LiteMap, as created in step 1a above;
      • $email = fill in a default user email address for LiteMap - this can be made up. This user account owns the default node and links types and acts as a template for all subsequent user accounts;
      • $password = fill in an default user password for LiteMap;
      • $email2 = fill in a theme admin user email address for LiteMap - this can be made up. This user account manages the LiteMap News list and is called the System Admin user.
      • $password2 = fill in an theme admin user password for LiteMap;
    2. Now run the adddefaultdata.php file on your server. You can do this in two ways:
      1. Run it from the command line. Note that the file will output data that you will need, specifically two UserIDs and the deafult role group ID (as well as possible error messages). You can get the user ids you will need from the database Users table, adnd the deafult NodeTypeGroupID from the NodeTypeGroup table if required. But I would send the output to a file myself.
      2. Surf to the file through your browser. In this case the three ids you need for your config file will be displayed on screen as well as any error messages.
      When it completes (hopefully without error) make a note of the ids it displays. You will need these in a moment to put in the Config file.
  3. Copy /config-sample.php to /config.php. You will need to edit the config.php file here to just add two things:
    1. $CFG->basedomain - this is the domain that is the default one for the installation. If you are running this as a single site this will be your actual domain. For multi-sites you still need a base domain entered here. The details config.php file for this base domain can be found in 'sites/default/'.
    2. $CFG->dirAddress - this is the file path on your server to the website code, ending in a '/'
  4. Copy sites/default/config-sample.php to sites/default/config.php.
    Edit settings in sites/default/config.php to point to your set up and database and to configure other important aspects of LiteMap.
    For the setting $CFG->ADMIN_USERID, $CFG->defaultUserID and $CFG->defaultRoleGroupID you need to copy in the ids given from step 2c. above
    All config file settings are explained below in Config Settings.
    The ones in red represent the minimum settings that you must look at and edit as required.
  5. At this point you should be able to surf to the website and check it comes up OK.
  6. If all is well, remove the install folder from your website.

Config Settings

All setting have comments explaining their use.

$CFG->privateSite = false;
( Does this site require login for all access, including interface and API; true if it does, else false if it does not. This basically would lock down the site for access by non-logged in users. You would need to really edit the login page to make it more of a landing page if you choose to set this to 'true', as the login page would be what people initally always see. ).

$CFG->START_DATE = mktime(0, 0, 0, 01, 01, 2013);
( This is the date you setup your site. It is currently used by the site Analytics. E.g. mktime(0, 0, 0, 7, 01, 2012) = 1st July 2012 ) just edit the last three number month, date, year).

$CFG->SITE_TITLE = "The title of your instance of LiteMap";
( This is used in the header title as well as being displayed to the user in various places. )

$CFG->ADMIN_USERID = "the database user id number of your System administrator";
( This person is the owner of the News nodes in the system and you have to log in as them to manage themes. You will have been given this id as output from the install process. Please note: by default News is not currently active on the homepage. So you would need to uncomment the News section to see andy News entriues you add through the 'admin' area.)

$CFG->defaultUserID = "the database user id number of your Default user";
( This person acts as a template for new users. Their node and link types are copied into new user accounts. You will have been given this id as output from the install process.)

$CFG->defaultRoleGroupID = "";
( As found in the database NodeTypeGroup table. You will have been given this id as output from the install process)

  1. Go to https://developer.apps.yahoo.com/dashboard/createKey.html and create a project.
  2. Fill out any required fields such as the Application name and description.
  3. Set the 'Applcation Type' to be 'Web-based'.
  4. Set the 'Home Page URL' to be you hub url
    It should match with the current domain given in $CFG->homeAddress minus the final slash.
  5. Int 'Access Scope' select 'This app requires access to private user data'.
    A section will expand out. Enter the Application Domain as in 4. above.
  6. In 'Select APIs for private user data access:', select 'Social Directory' and leave it as 'Read Public'.
  7. Once you have created the project, copy and paste the 'Consumer Key' as the $CFG-SOCIAL_SIGNON_YAHOO_ID
    and the 'Consumer Secret' as the $CFG->SOCIAL_SIGNON_YAHOO_SECRET
    and set $CFG->SOCIAL_SIGNON_YAHOO_ON = true.

Language

The current language system allows you to do two important things:

  1. Add another language: You can change the language you are using by translating all the language files into another language.
  2. Site specific text: You can change the text in your version of the hub to be site specific.

Adding another language

If you want to add a new language set to the LiteMap you need to:

  1. Create a new folder inside the 'language' folder whose name the two letter abbreviation for the language you are providing a translation for. These codes are specified by the ISO 639 standard, e.g.

    Language Code Language
    en English
    fr French
    it Italian
    de German

     

  2. Copy the language files from inside the default 'en' language folder into your new folder.
  3. Inside any given language file, each item is set up as a key/value pair, in the form key='value'. You must translate/change only the values on the right hand side, NEVER change the key name on the left.
  4. Change the language variables in the config file: $CFG->language = 'en'; to reflect your new language of choice. The $CFG->language setting must match the folder name for your new language.

NB: Some HTML is mixed up in the language files especially in the language heavy areas like the About and the Help pages.

The languagecore.php file contains the node and link names used by other language files. It will always be loaded first so that other language files can refernece it.

Site specific text

To provide site specific text for any of the interface languages you follow this basic pattern:
  1. Locate the term you wish to replace in the language files.
  2. Create a file with the same name as the file containing the term you wish to change and put it in the languages folder of sites/default or sites/multi/, in the same structure e.g. sites/default/language/en/languageui.php.
  3. Inside this new file you must use the same key label on the left as in the original file, and then just replace the part on the right after the '=' sign. All domain specific files in the 'sites' language sub-folders are loaded after the main language files, and provided you have used the same keys, the text will be replaced with your version when the language files are loaded.
NOTE: If what you want to change any of the core terms, the names of the categories or links, for example, if you wish to call 'Themes' say 'Topics' instead on your Hub, you can just create a custom languagecore.php file and change the text in that. These terms will ripple through the rest of the interface text as all the rest of the text should reference those terms through their keys. This makes it easy to change the basic terminology for a given Hub community without editing lots of interface text.

IMPORTANT: There are two none-standard items in the language folder that are not handled in the same way as the other language files:

  1. The countries.php file: This holds the country names used in the location drop-down menus displayed to users. It is also the place where you set your default country when displaying the list ($LNG->DEFAULT_COUNTRY=). When translating these, please, as with the other files change only the part on the right after the '='. NOTE: this file may need updating if new countries come into existence. The location of where to check for updates is in the file header.
  2. The mailtemplates folder: The files inside this folder are used when constructing the emails send out to users. When translating them please change only the words and leave all HTML alone, unless you know what you are doing. Be especially careful about leaving all %s items alone. These are replace with specific text when an email is created. So the number of %s and their order in the file is very important. Please don't remove any or that email will no longer be written correctly.


Tailoring your LiteMap

At present, if you want to change the look and feel or text of your LiteMap site,these are the main places you should look at making changes:
  • Themes: The style sheets for the ui can be found in 'themes/default/styles/'. Please don't edit those ones. If you want to create your own theme you should copy the 'default' folder, rename it and edit the styles there. In the images folder you will see some image files like the header logo etc, that you will also probably want to replace. Just make sure the files names and sizes stay the same. In addition you can override themes for a given domain or the default domain in the 'sites' folder. For more information see the README.txt file in the 'themes' folder.
  • Language: All text displayed in LiteMap has been placed in various files in the 'language' folder system. So you can override all the default texts your specific site needs. You may especially like to add to or redo the about page (language/en/about.php and help.php pages). (See the Language section of these notes for more details and there is also a README.txt file in the 'language' folder).
  • Homepage: The file that draws the homepage (ui/homepage.php) is in the 'ui' folder. While all the text can be changed through the language files, you may want to completely redo this page for your specific version of LiteMap.
  • Images: The theme specific images can be found in the themes as mentioned above. All other icons and images can be found in the 'images' folder. If you want to replace an image make sure the size and file name stay the same and you will not need to edit any code. We suggest you backup the images folder before you make changes.
  • Headers and Footers: The header and footer files can be found in the 'ui' folder. The main 'header.php' and 'footer.php' do the base site pages, but you will see there are also specific header and footer pages for other parts of the site, like popup dialogs and the embeds etc., so you may want to look at all of them when making any changes.


Additional Developer Notes

RESTful API

LiteMap has a RESTful API. This returns jsonld formatted data in the Catalyst Interchange Format.
The main use patterns are:

'your website address/api/views/map id number' - for getting maps
'your website address/api/conversations/group id' - for getting all maps in a group

There is a jsonld icon button on group and map pages to get the api url for that group or map. The code handling these requests can be found in 'api/restapi.php'. Note also there is a .htaccess file in the 'api' folder which directs between the RESTful api calls and the ui RPCs

Database Schema

As well as the install/mysql.sql file which has the sql for the database schema, there are two additional documents:

  • 'docs/LiteMapSchemaDocs.htm', which describes the schema in an easy to read format.
  • 'docs/LiteMapDatabaseSchemaDrawing.pdf', which shows a graphics model of the LiteMap database schema and how the tables relate.

Datamodel Rules

If you are interested in the connection rules for the Datamodel that LiteMap uses these are listed in 'docs/LiteMapDatamodelConnectionRules.pdf'

README

In addition to this document many folders in 'src' have a README.txt file with specifics about areas like 'sites', 'sites/multi', 'sites/default', 'sites/default/custom' etc..
